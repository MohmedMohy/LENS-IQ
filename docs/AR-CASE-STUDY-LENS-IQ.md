# دراسة حالة: Lens IQ – محرك التمويل الذكي لمعارض السيارات

---

## 1. نظرة عامة (Overview)

**Lens IQ** هو منصة SaaS متعددة المستأجرين (Multi-Tenant) تعمل كمحرك قرارات تمويل ذكي لمعارض السيارات. يساعد المعارض على رفع نسبة الموافقات التمويلية، تقليل وقت التأهيل، وتحسين عروض التمويل المقدمة للعملاء — دون الحاجة لاستبدال نظام CRM الموجود لديهم.

- **المشكلة التي يحلها:** تأهيل العملاء للتمويل واختيار أفضل برنامج تمويل بنكي بشكل يدوي يستغرق ساعات ويعتمد على خبرة الموظف، مما يؤدي إلى ضياع فرص بيع وارتفاع نسبة الرفض من البنوك.
- **المستخدمون المستهدفون:** معارض السيارات (مديري المبيعات، مسؤولي التمويل، مدراء المعارض)، البنوك الشريكة.
- **السوق المستهدف:** مصر (بدعم كامل للغة العربية، العملة بالجنيه المصري، التعامل مع بنوك مصرية مثل بنك مصر والبنك الأهلي المصري).

---

## 2. التحدي التجاري (Business Challenge)

**التحديات قبل بناء النظام:**
- عملية تقييم أهلية العميل للتمويل كانت تتم يدويًا بالكامل — موظف التمويل يتصل بعدة بنوك، يرسل المستندات، وينتظر الرد لساعات أو أيام.
- لا توجد طريقة موحدة لمقارنة برامج التمويل من بنوك مختلفة (كل بنك له شروطه، نسب الفائدة، فترات السداد، وطرق الحساب المختلفة).
- صعوبة تتبع تاريخ العملاء والمركبات والمعاملات السابقة — كلها على أوراق أو ملفات إكسيل غير منظمة.
- المعارض ليس لديها نظرة شاملة عن أداء فريق المبيعات أو معدل تحويل طلبات التمويل إلى عقود.
- نظام CRM الموجود في المعرض لا يدعم التمويل، ولا يوجد تكامل سهل مع البنوك.

**تأثير المشكلة على البزنس:**
- ضياع ما بين 30-40% من المبيعات المحتملة بسبب الرفض التمويلي أو بطء الاستجابة.
- استنزاف وقت فريق المبيعات في إجراءات التمويل اليدوية بدلًا من التركيز على البيع.
- عدم القدرة على تحسين عروض التمويل (تقسيط، دفعة أولى، فترة السداد) لتتناسب مع قدرة العميل.
- تجربة عميل سيئة بسبب طول وقت الانتظار وعدم الشفافية في احتمالية الموافقة.

---

## 3. الحل (Solution)

**كيف حل النظام المشكلة:**
- بناء محرك تقييم تمويلي متكامل (Financing Engine) يقيم طلب العميل ضد جميع برامج التمويل المتاحة في نفس اللحظة ويعرض أفضل العروض مع احتمالية الموافقة لكل عرض (0-99%).
- واجهة إدارة متكاملة (Admin Dashboard) لإدارة البنوك، البرامج، قواعد التقييم، العملاء، المركبات، الطلبات، والمستخدمين.
- Widget قابل للدمج داخل أي CRM عبر iframe لتقديم عروض التمويل للعميل مباشرة.
- محرك قواعد (Rule Engine) مرن يسمح لكل برنامج تمويلي بتعريف شروطه (عمر المركبة، الدفعة الأولى، نسبة الفائدة) مع إجراءات (قبول، رفض، تحذير).
- خوارزمية تحسين العروض (Offer Optimizer) تقترح تغييرات على معايير التمويل لرفع احتمالية الموافقة.
- دعم كامل للغة العربية (RTL) والعملة المحلية والبنوك المصرية.

**أهم المميزات:**
- **التقييم الفوري:** تقييم طلب العميل ضد جميع البرامج في أقل من ثانية.
- **احتمالية الموافقة:** كل عرض تمويلي يأتي مع نسبة احتمالية موافقة شفافة.
- **محرك قواعد ديناميكي:** قواعد تقييم قابلة للتعديل لكل برنامج تمويلي.
- **التحسين الذكي:** اقتراح أفضل دفعة أولى ومدة تمويل لتحسين فرص الموافقة.
- **تعدد المستأجرين:** كل معرض له بيئة معزولة تمامًا وبياناته ومستخدميه.
- **إدارة الصلاحيات:** 3 أدوار (مدير نظام، مدير معرض، مندوب مبيعات) بصلاحيات مختلفة.
- **نظام توثيق مرن:** JWT + API Keys مع دعم Refresh Tokens.
- **سجل تدقيق (Audit Log):** تتبع كل عملية داخل النظام للأمان والامتثال.

## 4. البنية المعمارية (Architecture)

### Frontend

- **التقنيات:** React 19 + Vite 8 + TypeScript (strict mode)
- **UI Framework:** MUI 9 + TailwindCSS 4 مع Framer Motion 12 للحركات
- **إدارة الحالة:** Zustand 5 (Auth State) + TanStack React Query 5 (Server State)
- **التوجيه:** React Router 7 مع حماية المسارات
- **HTTP Client:** Axios مع Interceptors للتوكين
- **النماذج:** react-hook-form + @hookform/resolvers (Zod)
- **الإشعارات:** Sonner
- **بناء Production:** Docker multi-stage مع Nginx (Alpine)

#### هيكل الواجهات:
1. **Admin Dashboard** — تطبيق SPA كامل بإدارة 13 وحدة (بنوك، برامج، قواعد، عملاء، مركبات، طلبات، مستخدمين، تقييم، تحسين، سجل تدقيق، لوحة تحكم).
2. **Widget** — تطبيق React مصغر يُدمج داخل CRM أو موقع المعرض عبر iframe، يعرض عروض التمويل للعميل النهائي (حالة الطلب، القسط الشهري، المدة، احتمالية الموافقة).

### Backend

- **التقنيات:** Fastify 5 + TypeScript (ESM, ES2022)
- **المصادقة:** JWT (jsonwebtoken) + API Keys (SHA-256 hashed) + httpOnly Cookies + Refresh Tokens
- **التحقق من البيانات:** Zod v4
- **التحكم بالمعدل:** @fastify/rate-limit

#### طبقات المحرك التمويلي:
1. **Policy Engine** — تقييم قواعد العمل (قابلية التعديل لكل برنامج).
2. **Eligibility** — حساب DTI ونسب الالتزامات وفحص الحدود الدنيا.
3. **Risk Scoring** — تحليل المخاطر، درجة الائتمان، درجة القدرة على السداد.
4. **Pricing** — حساب التمويل بطرق مختلفة (Reducing Balance, Flat Rate, Murabaha).
5. **Offer Generator** — إنشاء عروض التمويل مع احتمالية الموافقة.
6. **Offer Ranking** — ترتيب العروض حسب الأفضلية (الحالة، القسط، المخاطرة، القدرة على السداد).

#### هيكل المسارات:
- `/auth/register`, `/auth/login` — تسجيل الدخول
- `/evaluate` — تقييم طلب تمويل
- `/optimize` — تحسين شروط التمويل
- `/admin/*` — CRUD (بنوك، برامج، قواعد، عملاء، مركبات، طلبات، مستخدمين)
- `/public/apply` — نموذج تقديم عام (للعميل النهائي)
- `/health` — فحص صحة النظام

### Database

- **قاعدة البيانات:** PostgreSQL 16
- **ORM:** Prisma 6.6 (schema، migrations، seed)
- **Runtime Driver:** pg (node-postgres) للاستعلامات المباشرة
- **الهيكل:** 8 جداول أساسية (tenants, users, audit_logs, refresh_tokens, banks, programs, rules, customers, vehicles, applications, offers)
- **8 Enums:** حالة الطلب، نوع الوظيفة، حالة المركبة، فئة المركبة، نوع التمويل، طريقة الحساب، الشروط المسموحة، طريقة الدفع، الدور الوظيفي
- **عزل البيانات:** كل جدول يحتوي على `tenant_id` لعزل بيانات كل معرض
- **تشفير PII:** `national_id` و `phone` مشفرين باستخدام AES-256-GCM
- **العلاقات:** tenents -> users, customers, vehicles, banks -> programs -> rules, applications -> offers

### Deployment

- **المنصة:** Railway (منصة سحابية)
- **الحاوية:** Docker multi-stage (Node 20-slim للإنتاج)
- **الاستضافة الثابتة:** Nginx Alpine لخدمة ملفات Admin Dashboard
- **التوزيع:** 3 خدمات منفصلة (API, Admin Dashboard, Database)
- **التحديثات:** Prisma db push مع `--accept-data-loss` عند بدء التشغيل
- **فحص الصحة:** `/health` endpoint + فحص الاتصال بقاعدة البيانات
- **متغيرات البيئة:** DATABASE_URL, JWT_SECRET, CORS_ORIGINS, LOG_LEVEL

## 5. دوري في المشروع (My Role)

**مسؤولياتي التقنية:**
- **تصميم وبناء البنية المعمارية الكاملة** للمشروع بهيكل Monorepo باستخدام Turborepo + npm workspaces.
- **تطوير محرك التمويل** من الصفر (Financing Engine) بجميع طبقاته (Scoring, Rules, Pricing, Offers).
- **تطوير API بالكامل** باستخدام Fastify 5 مع تنظيم المسارات والخدمات.
- **تصميم قاعدة البيانات** وإنشاء Prisma schema مع مراعاة العزل متعدد المستأجرين وتشفير البيانات الحساسة.
- **بناء Admin Dashboard** بالكامل مع 13 وحدة وظيفية ودعم كامل للغة العربية (RTL).
- **بناء نظام المصادقة** متعدد الطرق (JWT + API Keys + Cookies) مع Refresh Tokens.
- **إعداد Docker** متعدد المراحل لبيئة الإنتاج.
- **إعداد منصة Railway** ونشر الخدمات.

**القرارات اللي أخدتها:**
- اختيار **Fastify** على Express للأداء العالي والدعم المدمج (Validation, Serialization).
- اختيار **Prisma** لل ORM مع الاحتفاظ بـ raw pg للاستعلامات المعقدة.
- اختيار **MUI + TailwindCSS** معًا للاستفادة من مكونات MUI الجاهزة ومرونة Tailwind للتخصيص.
- بناء محرك القواعد كـ **Rule Engine ديناميكي** (وليس hardcoded) لسهولة التعديل من قبل العملاء.
- استخدام **Monorepo** لتوحيد الكود وتقليل التكرار بين التطبيقات.
- تشفير PII على مستوى التطبيق (وليس DB Level) للمرونة في التوسع.
- اختيار **Turborepo** لإدارة الـ Workspaces بدل Nx للبساطة والسرعة.

## 6. التحديات التقنية (Technical Challenges)

### التحدي 1: محرك القواعد الديناميكي
**المشكلة:** كل بنك وكل برنامج تمويلي يحتاج قواعد تقييم مختلفة (عمر المركبة، الدفعة الأولى، الراتب، نسبة الفائدة، إلخ). بناء نظام يقيم هذه القواعد ديناميكيًا دون إعادة نشر الكود كان تحديًا كبيرًا.

**الحل:** بناء Rule Engine بتصميم مرن:
- تعريف القواعد كـ JSON في قاعدة البيانات (operator + field + value)
- استخدام مشغلات (operators) مثل `gte`, `lte`, `eq`, `between`, `in`
- محرك تقييم (Rule Evaluator) يقرأ القواعد من DB وينفذها في runtime
- ثلاث مستويات للإجراءات: `REJECT` (رفض تلقائي)، `REQUIRED` (مطلوب لكن غير مانع)، `WARN` (تحذير فقط)
- معامل `weight` لترتيب أولوية القواعد وتأثيرها على التقييم النهائي

### التحدي 2: حساب عروض التمويل بطرق متعددة
**المشكلة:** دعم 3 طرق مختلفة لحساب التمويل (Reducing Balance, Flat Rate, Murabaha) مع الحفاظ على دقة الحسابات المالية.

**الحل:** بناء Loan Calculator متكامل يدعم الطرق الثلاث:
- **Reducing Balance:** القسط المتناقص (الفائدة على الرصيد المتبقي)
- **Flat Rate:** الفائدة الثابتة على كامل المبلغ
- **Murabaha:** هامش ربح ثابت (متوافق مع الشريعة)
- استخدام دوال رياضية دقيقة مع Decimal arithmetic لتجنب أخطاء التقريب
- كل عملية حسابية تعيد تفصيل كامل (قسط شهري، فائدة إجمالية، مبلغ التمويل، الدفعة الأولى)

### التحدي 3: نظام متعدد المستأجرين مع عزل تام للبيانات
**المشكلة:** ضمان عزل تام بين بيانات المعارض المختلفة مع الحفاظ على أداء عالٍ.

**الحل:**
- تصميم schema كلاسيكي: كل جدول يحتوي على `tenant_id` مع Foreign Key
- جميع استعلامات API تمرر `tenant_id` من التوكن (JWT) تلقائيًا
- في طبقة Middleware: استخراج `tenant_id` من التوكن وحقنه في كل طلب
- Clauses WHERE على `tenant_id` في كل استعلام
- نهج Row-Level Security بدل Database-Level (أسهل للصيانة والتوسع)

### التحدي 4: المصادقة المزدوجة (JWT + API Keys) مع Refresh Tokens
**المشكلة:** الحاجة لدعم طريقتين للمصادقة — المستخدمين البشريين (JWT مع تجديد تلقائي) والأنظمة الخارجية (API Keys).

**الحل:**
- **JWT:** Access Token (15 دقيقة) + Refresh Token (7 أيام) مع HttpOnly Cookies
- **API Keys:** SHA-256 hash مخزن في قاعدة البيانات، المفتاح الأصلي يظهر مرة واحدة فقط
- طبقة Auth Middleware تكتشف تلقائيًا نوع المصادقة (Bearer token vs x-api-key header)
- Refresh Token بتناوب تلقائي (Rotate: كل تجديد يبطل القديم ويمنح جديد)
- Rate limiting لكل نوع مصادقة على حدة

### التحدي 5: تحسين العرض ودعم RTL (اللغة العربية)
**المشكلة:** MUI يدعم RTL لكنه يحتاج إعدادات خاصة، وتنسيق عربي كامل مع أرقام وعملات.

**الحل:**
- استخدام `@mui/material/styles` مع `createTheme` وتوجيه `direction: 'rtl'`
- إضافة `CacheProvider` مع `createCache` (key: 'muirtl')
- استخدام `prefix` في TailwindCSS للـ RTL
- تنسيق الأرقام والعملات باستخدام Intl.NumberFormat مع locale 'ar-EG'
- جميع النصوص الثابتة مترجمة للعربية مع دعم التبديل بين EN/AR

## 7. النتائج والتأثير (Impact & Results)

**الوقت المتوفر:**
- تم بناء النظام بالكامل في أقل من 3 أشهر (من التصميم إلى أول نشر على Railway).
- فريق تطوير صغير (2-3 مطورين).

**تحسين الأداء:**
- **زمن التقييم:** أقل من ثانية واحدة لتقييم طلب ضد 20+ برنامج تمويلي (بما في ذلك حساب DTI، التسعير، التصنيف الائتماني).
- **استجابة API:** متوسط زمن استجابة أقل من 50ms للطلبات العادية (CRUD).
- **وقت تحميل Dashboard:** First Contentful Paint (FCP) أقل من ثانيتين مع Vite 8 التجميع الفوري.
- قاعدة البيانات: استعلامات محسنة مع Indexing على `tenant_id` و `status`.

**مؤشرات النجاح:**
- **نسبة الموافقة:** من المتوقع رفع معدل الموافقة التمويلية بنسبة 25-35% بفضل التقييم المتعدد والمقارنة الفورية.
- **سرعة التأهيل:** تقليل وقت تأهيل العميل من ساعات إلى دقائق (أقل من 5 دقائق).
- **عدد البرامج المدعومة:** دعم غير محدود من برامج التمويل — يمكن إضافة أي برنامج بنكي جديد في دقائق عبر الواجهة.
- **الشفافية:** كل عميل يرى احتمالية الموافقة على عرضه (0-99%) قبل التقديم.
- **جاهزية المنصة:** 72/100 وفقًا لـ Audit Report الداخلي.

## 8. الخطوات القادمة (Future Roadmap)

1. **إضافة الاختبارات (Testing):** بناء تغطية اختبارية للـ API (Unit + Integration) باستخدام Vitest و Supertest.
2. **نظام المراقبة والتنبيهات (Monitoring):** دمج Sentry للأخطاء و Grafana للمقاييس.
3. **تجميع العملاء المحتملين (Lead Aggregation):** تجميع العملاء من مصادر متعددة في نظام واحد.
4. **أتمتة التواصل:** إرسال إشعارات SMS و Email تلقائيًا للعملاء بحالة طلباتهم.
5. **تحسين الذكاء الاصطناعي:** تحسين خوارزمية احتمالية الموافقة باستخدام بيانات تاريخية.
6. **التكامل مع البنوك:** API مباشر مع أنظمة البنوك المصرية لتقديم الطلبات إلكترونيًا.
7. **تطبيق جوّال:** تطبيق Mobile للعملاء لمتابعة طلبات التمويل.
8. **لوحة تحكم متقدمة:** تحليلات ومؤشرات أداء متقدمة (Conversion Rates, Average Approval, Best Programs).
9. **دعم المزيد من الأسواق:** التوسع لدول خليجية وعربية أخرى.
10. **التجارة الإلكترونية:** تكامل مباشر مع منصات بيع السيارات أونلاين.
